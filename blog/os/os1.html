To understand more fully the operating system’s role, we next explore operating systems from two viewpoints:

1. User View

2. System View

User View

For normal computer user the operating system is designed mostly for ease of use, with some attention paid to performance.

In other cases, users sit at workstations connected to networks of other workstations and servers. These users have dedicated resources at their disposal, but they also share resources such as networking and servers,including file, compute, and print servers.

Some computers have little or no user view. For example, embedded computers in home devices and automobiles may have numeric keypads and may turn indicator lights on or off to show status, but they and their operating systems are designed primarily to run without user intervention.

System View

From the computer’s point of view, the operating system is the program most intimately involved with the hardware. we can view an operating system as a "resource allocator".

A computer system has many resources that may be required to solve a problem: CPU time, memory space, file-storage space, I/O devices, and so on. The operating system acts as the manager of these resources.

An operating system is a control program. A control program manages the execution of user programs to prevent errors and improper use of the computer. It is especially concerned with the operation and control of I/O devices.

Since bare hardware alone is not particularly easy to use, application programs are developed. These programs require certain common operations, such as those controlling the I/O devices. The common functions of controlling and allocating resources are then brought together into one piece of software: the operating system.



Single-Processor Systems

On a singleprocessor system, there is one main CPU capable of executing a general-purpose instruction set, including instructions from user processes.

Almost all singleprocessor systems have other special-purpose processors as well. All of these special-purpose processors run a limited instruction set and do not run user processes.

If there is only one general-purpose CPU, then the system is a single-processor system.

Multiprocessor Systems

Within the past several years, multiprocessor systems (also known as parallel systems or multicore systems) have begun to dominate the landscape of computing.

Such systems have two or more processors in close communication, sharing the computer bus and sometimes the clock, memory, and peripheral devices. Multiprocessor systems first appeared prominently appeared in servers and have since migrated to desktop and laptop systems.

Symmetric multiprocessing architecture


In Symmetric multiprocessing architecture each processor performs all tasks within the operating system. SMP means that all processors are peers; no boss–worker relationship exists between processors.

A recent trend in CPU design is to include multiple computing cores on a single chip. Such multiprocessor systems are termed multicore.

A dual-core design with two cores placed on the same chip.

These can be more efficient than multiple chips with single cores because on-chip communication is faster than between-chip communication. In addition, one chip with multiple cores uses significantly less power than multiple single-core chips.

It is important to note that while multicore systems are multiprocessor systems, not all multiprocessor systems are multicore

Hardware Interrupts

Please note that pressing any key is not a Hardware Interrupt it is just a Mechanical Action.

  • Hardware devices(Keyboard, mouse, Hard-Disk, etc.) can raise "Hardware Interrupts"
  • Basically Hardware Interrupts means creating an electrical signal on some connection to CPU(/Bus)
  • This is notified to CPU(in Hardware)

Now CPU's normal execution is interrupted!

  • CPU will not continue doing the fetch, decode, execute, change PC cycle!

  • What happens then?
    • On hardware interrupt, the PC changes to a location pre determined by CPU manufaturers!
    • Now CPU resumes normal execution.

Now question arises is that what is there at this pre-determined address?

---> It is OS! , How's that? will see!

Let's understand Boot process again!

1. BIOS runs "automatically" because at the initial value of PC(On power ON), the manufacturers have put in the BIOS code in ROM. now CPU is running BIOS code.

2. BIOS looks up for boot loader(on the default boot device) and loads it in the RAM, and passes control over it. now CPU is running boot loader code.

3. Boot loader gets boot options from human user, and loads the selected OS in memory and passes over to OS. now CPU is running OS code.

Now we will see answer to our unanswered question.

How OS is there at pre-determined address?

When OS starts running initially, it copies relevent parts of it's own code at all possible memory addresses that hardware interrupt can lead to!

Now whenever there is hardware interrupt the PC will change to predetermined location and control will jump into OS code. generally, this is called as "taking control of Hardware"

Multiprogramming

A system where multiple processes(!) exist at the same time in the RAM(But only one runs at a time!)

Program

Just a binary (machine code) file lying on the hard drive. E.g. /bin/ls and remember Does not do anything its own!(Executable file)

Process

It is a program that is executing

A program must exist in "RAM" before ir executes!

One program can run as multiple processes.(It means that suppose, you open "vi editor" and in new tab u open one more vi editor).

Multi-tasking

Time sharing between multiple processes in a multi-programming system.(The OS enables this!)

Point to be noted is if system is multiprogramming then it is possible that it will not be an multitasking! but if system is multitasking then the it has to be multiprogramming!

Some requirements Multi-tasking

Two processes should not be able to steal time of each other.

Two processes should not be able to see data/code of each other.

Two processes should not be able to modify data/code of each other.

Now Let’s understand What kind of Hardware, OS interplay makes Multitasking possible


There are two types of CPU instructions

1. Normal instructions like MOV, JMP, ADD etc.

2. Privileged instructions these are normally related to hardware devices. for example, IN, OUT(write to I/O memory locations). INTR(software interrupt)

There are two modes of CPU operation

1. CPUs have a mode bit (can be 0 or 1)

2. The two values of the mode bit are called: User mode and Kernel mode

3. If the bit is in user mode Only the normal instructions can be executed

4. If the bit is in kernel mode Both the normal and privileges instructions can be executed

5. If the CPU is “made” to execute privileged instruction when the mode bit is in “User mode” It results in a particular hardware interrupt(What should happen on a hardware interrupt now?, answer for this is above :)

  • The opearting system code runs in kernel mode. How? Will see!
  • The application code runs in user mode How? will see!
  • So application code can not run privileged hardware instructions.
  • Transition from user mode to kernel mode and vice-versa is done with Special instruction called “software interrupt” instructions.(E.g. INT instruction on x86)

Software interrupt instruction

The INT on x86 processors does two things :

1. Changes mode from user mode to kernel mode in CPU.

2. Jumps to a pre-defined location! Basically changes PC to pre-defined value.(Now you now that what is there at this pre-defined value, obiviously OS :))

Now question is arises that who runs this INT instruction(Your application will run this!)

What’s the use of these type of instructions?

  • An application code running INT 0x80 on x86 will now cause
    • Change of mode
    • Jump into OS code
  • This is effectively a request by application code to OS to do a particular task!
  • E.g. read from keyboard or write to screen.
  • OS providing hardware services to applications!
  • A proper argument to INT 0x80 specified in a proper register indicates different possible request

In the above diagram when there is user process is executing, On CPU there is only user program is running nothing else is running on the CPU.

Whenever the user program does the system call(here consider INT 80 instruction!) , its a software interrupt and mode of the CPU is changes from "user mode" to "kernel mode".

The particular OS code that runs in kernel mode is called as system call!

Now in kernel mode when system call does INRET instruction it changes mode-bit to 1 and enters into a user mode.

How does application code run INT instruction?

C library functions like printf(), scanf() which do I/O requests contain the INT instruction!

Control flow : Application code -> printf -> INT -> OS code -> IRET -> back to printf code -> back to application code

The OS code is sitting in memory and runs intermittantly when a "Software" or "Hardware" interrupt occurs


Now let's see how multi-tasking actually happens

  • Multi-tasking is enabled using "Timer interrupt"(Setting timer register is a privileged instruction)
  • OS sets timer interrupt and “passes control” over to some application code. Now only application code running on CPU !
  • When time allocated to process is over, the timer interrupt occurs and control jumps back to OS (hardware interrupt mechanism)
  • The OS code that runs here (the ISR) is called “scheduler”
  • OS can now schedule another program

conformance testing of the system calls that is done by calling the different system call functions (e.g. write(), read(), open(), etc.) and checking if they do the desired job.

Conformance testing of an OS:

OS provides "system calls" to an applications. E.g. printf(), scanf(), etc. results in an System call.

For example:

1. By executing "int 0x80" instructions, 2. By specifying different arguments in register(e.g. EAX) the "int 0x80" may jump to different locations. Each of these then acts as a different System call.
