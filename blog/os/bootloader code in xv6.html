Bootloader code in xv6

lets understand again what does exactly bootloader do!

1. Bootloader itself is loaded by the BIOS at a fixed loaction in memory and BIOS makes it run.

2. why is it at fixed loaction?

• Because the BIOS code is written by the CPU manufaturer. (BIOS code itself is located at fixed loaction)

The BIOS code look for bootloader code on the boot device in sector 0 ,when it has copy paste it in memory that has to be a fix location(these loactions are normally documented in manuals given by CPU manufaturer)

Our job as OS programmer is to write the bootloader code.

Bootloader pick up code of OS from known location and loads it in memory and makes the OS run.

XV6 "bootloader code" is in "bootasm.S" and "bootmain.c"(let us see makefile)

Makefile

From here the kernel is getting built...

Line number "93" : builds disk image of XV6

• In this the first "dd" command : "if" for "input file" and "of" for output file, it simply reads zeros from /dev/zero file (which is the infinite source of zeros), writes 10000 blocks into xv6.img file (each block is typically 512 byte)

• "dd if=bootblock of=xv6.img conv=notrunc" in this reads from bootblock file and write into initial part of xv6.img file without truncating.

• "dd if=kernel of=xv6.img seek=1 conv=notrunc" in this reading from kernel file and writing into xv6.img skiping 1 block i.e. is after 512bytes, without truncating

so overall this part of code simply concatenates "bootblock cod0e" and "kernel code", followed by zeros....

Now let us move to line number 103

We can see that bootblock is consisting of "bootasm.S" "bootmain.c"

• line number 104 and 105 indicating that "bootmain.c" and "bootasm.S" is compiled using cc and obtains "bootmain.o" and "bootasm.o" file respectively...

• now after this linker comes into a picture i.e. "LD command" and it links bootmain.o and bootasm.o file and creates bootblock.o file...

In this command the interesting input to the linker is "-Ttext 0x7C00" this telling to the linker that you create the object code file but int the object code file you write "-Ttext 0x7C00" that when this code is loaded in the memory the code should get loaded at this location in memory this address is documented by processor manufaturer telling that we will entertain bootloader code at this location (7C00)

Now let's skip the OBJDUMP line

• The next line "$(OBJCOPY) -S -O binary -j .text bootblock.o bootblock" this is just copying bootblock.o as bootblock which is used at line number 93 in code...

So now what we are supposed to do is that understand the code in "bootasm.S bootmain.c" because these two files makes bootblock.

so let's see bootasm.S file...

bootasm.S

So this is a bootloader code,

Please note that whenver you see directives which begins with "." for example here ".code16 .globl" these are basically the additional instructions to the assembler to assemble i.e. convert to machine code in a particular way.

After that the "start" is just lable whenver there is call or jmp instruction the code will begin at this lable..

So the first line to be assembled is "cli". which is going to "disable interrupts".(So that until your code loads, all hardware interupts handler, no interrupts will occur)

While executing above piece of code,

• The IP will have value 7C00 so instruction "cli" will at 7C00.

• And the next instruction i.e. "xorw %ax,%ax" will be at "7C00 + 4" and so on... (IP gets incremented automatically)

• "xorw %ax,%ax" this instruction will XOR the register with itself and makes register zero(plz note that %ax is a 16-bit register)

• After that there are three instructions which makes all segment register zero..

What about code-segment?

• The code seg must have been initialized by BIOS to have a particular value thats why code segemnt is not being touched here..

Processor starts in real mode

• It works like 16-bit 8088

• Eight 16-bit generel purpose registers.

• Segment registers %cs, %ds, %es and %ss (The additional bits necessary to genrate 20-bit memory address from 16-bit registers i.e. addr = seg*4 + addr)

Relation between logical and physical address

• CPU is running particular instruction after that instruction is over CPU is going to send on the outside bus which goes to the memory and address(the address from where the next instruction is going to fetch!)

• IP points to the instruction address..

Bootloader: address translation

• The x86 processor does two types of memory management viz. paging and segmentation(it is possible for OS programmer to enable either paging, segmentation or both together)

• Right now we will just go with the fact that there is something called paging and segmentation in the "hardware" which to be configured by OS code therefore it has to to configured by bootloader (otherwise you can not execute any code!)

• xv6 configures the segmentation hardware to translate logical to linear addresses without change so that they are always equal.(Segmenatation is practically off)

• Once paging is enabled, the only interesting addresses mapping in the system will be linear to physical...

bootasm.S

This piece of code exist in all bootloaders for some historical legacy purposes we know that Seg::off is a 16-bit calculation(segment itself is a 16-bit with this we can calculate 20-bits of memory i.e. 2^20(1MB) of memory),after 2^20 ,8086 wrapped addresses to zero.

• 80286 introduced 21st-bit of address. but older software required 20-bits only. BIOS disabled 21st bit and some OS needed 21st bit so they enable it.

• So all the above piece of code is to enable the 21st bit..

• "inb $0x64,%al" #write to port 0x64

• "0x60" is keyboard controller.

• 21st bit belong to keyboard controller.

bootasm.S

Moving further, In this piece of code the first line is "lgdt gdtdesc" which means load gdt(global descriptor table(part of memory mangement unit)) so with this line we going to load data(given by "gdtdesc") to gdt.

When you scroll little down you find this "gdtdesc" which is refering to ".long gdt" right above you will find "gdt".

bootasm.S

What is it in this gdt?, It is an sequence of three numbers (Refer below image)

when you open "asm.h" file you will the definations of these numbers. refer the above image.

asm.h

So with "lgdt gdtdesc" we are going to load above three numbers to "gdt"

After "lgdt" there is 3 lines of code

• "movl %cr0, %eax" , moves %cr0 register to register %eax.

• "orl $CR0_PE, %eax" %eax is ored with "$CR0_PE". ($CR0_PE is a number) with this the particular bit is enabled in the %eax

• "movl %eax, %cr0" enabled bit is moved back to the %cr0 reg.

With these four lines of code the processor changes from 16-bit mode 32-bit protected mode. (real mode to protected mode).

The "real mode to protected mode" are totally unrelated to "user-mode and kernel-mode". (user-mode and kernel-mode are now exist in protected mode).

Real mode vs Protected mode

• In real mode we can access 16-bit reg and in protected mode we can access 32-bit mode

• Can accesss upto 2^32 memory

• Can do arthmetic in 32 bits

• The calculations of address which were happening using "seg*4+off" is no more going to happen

• The value in the seg register is now used as index into segment descripter table.

Segments in protected mode

• "gdt" is an array of base,limit and flags.

• The way in which the memory addresses are calculated by processor is like this, the value in the segments register is used as index in this gdt from there it will take base value and base value is added to offset to finally calculate addresse that goes to the RAM. the base value in this array is "zero" and and limit is 4gb (so when you add zero to offset you will get offset itself). so we are doing memory management setup where in every address issued by the CPU when it goes to memory management of CPU will basically remains the "same" because the segements are ineffevtive.

lets see the further code...

bootasm.S

With the first instruction we are copying SEG_KDATA left shifted to 3.

what is SEG_KDATA?

• In file "alltext" at line "0715" you will find the declaration of this macro!

alltext

so i.e. "2 'left shifted by' 3"!

After this the next three lines simply copies this value to all segments.. as a result of these three lines the ds,es and ss are pointed value 2..

And we are setting %fs and %gs to zero which we are not using right now!

By executing this "movl $start, %esp" line we are copying "7C00" to reg %esp.. and after that calling bootmain.(note that this will push the address of the whatever there is right after this call instruction , on to the stack.. and jump to the "bootmain")

now we jump into the "bootmain.c"(remember that bootblock is concatination of bootmain.o and bootasm.o that is bootmain.c is followed by bootasm.S)

bootmain.c

You will notice in this above piece of code "#include "elf.h"(gives structures to access elf file format),"struct elfhdr *elf;"(elfhdr is one such structure).if you notice line number 25 the "elf" pointer is pointing to 1 MEGA i.e. "0x10000".

Now there is function call of "readseg((uchar*)elf, 4096, 0);" function.

bootmain.c

You will notice that this function takes three arguments of which first one is "address in memory", 2nd one is count and 3rd one is offset.(So this function is reading from byte number zero upto 4096 bytes..)

Basically this function is doing is running loop in terms of sector size where sector size is 512 unit size and calling "readsect(pa, offset);" function. after doing some arthametic..

bootmain.c

• In this function the first call is to "waitdisk();".

• After that there is call to "outb(0x1F2, 1);".

so what is "outb(0x1F2, 1);"?

Open x86.h file you will find the defination this function..

x86.h

This is a basically c-function which has assebly code called "out".we know that in and out are io instructions they are used to write instruction into particaular location for the hardware controllers. they basically act as a bridge now between the software and hardware world. when you run in or out instr you are basically writing into the hardware controller and accessing the hardware devices. obiviously these are privillaged instructions.

By default we are running in the kernel mode now and running the out instruction.

so all these outb calls doing is they are instructing the hardware controller to read from the disk and make available data at particular memory location. at the end of all these io instructions the data will be rade by the hard-disk controller from the physical hard-disk and put into memory..

what has happend after these repeated calls of read sector function is that we rade the 4096 bytes from byte number zero of the hard-disk and put into address 1MB(why 1 MB beacuse we have declared elf as 1 MB).

so 1 MB location in the memory will contain 4k data from the first 4k byets of the hard-disk and outof the 4k first 512 were the bootloader and the next code was kernel.

now in the bootmain function there is call to entry() function which enters into kernel code now.
